# 工作原则：能力边界与结果质量

> 记录日期：2026-05-04
> 来源：零号自我反思

---

## 核心原则

**遇到能力或环境条件欠缺时，先解决欠缺的工具/环境问题，再继续工作。**
如果无法解决，必须如实告知用户，不要输出劣质结果。

---

## 细则

### 1. 能力边界判断清单

在开始任何任务前，先自问：

| 问题 | 如果"是" |
|------|----------|
| 我有合适的工具可以完整完成这项工作吗？ | 继续 |
| 文档格式是否有我无法解析的编码/格式问题？ | 先解决工具问题 |
| 依赖的环境是否已安装并可用？ | 先安装环境 |
| 是否存在我会用劣质结果糊弄用户的风险？ | 停止，如实汇报 |

### 2. 遇到无法完成任务时的正确处理流程

```
发现能力/环境不足
  → 判断：是否能快速解决（< 5分钟）？
      是 → 立即解决 → 继续工作
      否 → 判断：是否为用户明确要求的工作？
          是 → 如实告知用户缺少什么、需要什么
          否 → 跳过该部分，在回复中说明局限性
  → 绝对不允许：用不完整/乱码/劣质结果代替
```

### 3. 工具能力参考

| 任务 | 推荐工具 | 注意事项 |
|------|----------|----------|
| 读取 .docx | pandoc 或 docx-js | 注意编码问题 |
| 读取 .xlsx | openpyxl | - |
| 读取 .pdf | pdf skill | - |
| 写入中文文本文件 | qclaw-text-file skill | 避免编码乱码 |
| 写入 .docx | docx-js | 需 Node.js 环境 |
| 执行脚本 | PowerShell/Python | 注意 Windows 换行符 |

### 4. Word 文档处理能力现状（2026-05-04）

- **pandoc**：未安装，需手动安装
- **python-docx**：编码问题，中文显示乱码（Windows GBK 编码问题）
- **有效方案**：unpack XML → 手动解析或直接用用户提供的 .txt 版本

### 5. 质量检查标准

输出任何成果前自检：

- [ ] 内容完整吗？
- [ ] 格式正确吗？
- [ ] 编码是否正常？
- [ ] 用户能直接使用吗？
- [ ] 会不会给用户带来额外清理/修复工作？

如果任何一项为"否"，立即停止，重新评估。
