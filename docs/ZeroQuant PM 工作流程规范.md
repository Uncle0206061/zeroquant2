# ZeroQuant 2.0 PM 工作流程规范
文档版本：V1.0
作者：零号（PM）
适用周期：2026-05-05 起，贯穿 ZeroQuant 2.0 全生命周期

---

## 1. 文档目的

本规范定义 ZeroQuant 2.0 开发期间，PM（零号）与各开发终端之间的任务卡分发、反馈回收、审核确认、归档的全流程机制。

**核心原则**：
- 磨刀不误砍柴工 — Phase 1 开发启动前，所有任务卡一次性准备到位
- 减少交互摩擦 — 任务卡格式统一、反馈路径清晰，减少来回确认
- 先审后做 — 生成内容先让用户看，确认后再执行

## 3. Task Queue Skill 使用规范

### 3.1 安装路径
- 主版本：`Z:\ZeroQuant2\skills\task-queue\`（PM 管理）
- 各端本地：`~/.openclaw/skills/task-queue\`（各终端安装后本地副本）

### 3.2 目录结构
```
skills/task-queue/
├── SKILL.md               ← Skill 主文件
├── references/
│   ├── task-card-format.md  ← 任务卡格式规范
│   ├── review-checklist.md  ← 审核检查清单
│   └── feedback-template.md ← 反馈模板
├── scripts/
│   ├── write_file.py       ← 跨平台文本写入（含中文 BOM 处理）
```

### 3.3 任务卡格式规范（各端必须遵循）

每张任务卡必须包含以下字段：

```markdown
# 任务卡 TaskCard-{序号}
版本：v1.0 | 日期：YYYY-MM-DD | 来源：PM

## 基本信息
- 任务编号：TC-XXX-{终端}
- 任务名称：
- 任务类型：代码编写 / 配置搭建 / 接口开发 / 联调 / 文档 / 测试
- 截止时间：
- 前置依赖：[列出依赖的任务卡编号，完成后才能开始]

## 任务内容
[具体任务描述，清晰可执行]

## 交付物
- [ ] 交付物1（具体路径/格式）
- [ ] 交付物2

## 审核点
1. [审核条件1，可验证]
2. [审核条件2，可验证]

## 反馈要求
- 完成时：在 tasks/{终端}/done/ 写入反馈卡
- 异常时：立即在 tasks/{终端}/inbox/ 写问题描述，等 PM 回复
```

## 4. 任务卡生命周期

```
PM 创建任务卡
    ↓
写入 tasks/{终端}/inbox/（PM 端）
    ↓
各端 OpenClaw 轮询检测（每10分钟）
    ↓
各端下载任务卡 → 执行
    ↓
完成 → 写入 tasks/{终端}/done/{任务卡编号}.md
    ↓
PM 审核 → 通过/打回（打回需说明原因）
    ↓
通过 → 归档至 tasks/archive/
```

### 4.1 轮询机制
- 各端 OpenClaw 每 10 分钟轮询一次 `tasks/{终端}/inbox/`
- 发现新任务卡后，记录到 TASKS.md 开始执行
- 完成后写入 `done/` 并在 TASKS.md 更新状态

### 4.2 任务卡优先级
| 优先级 | 含义 |
|--------|------|
| 🔴 高 | 阻塞其他任务，必须优先完成 |
| 🟡 中 | 按时完成即可 |
| 🟢 低 | 可在空闲时间完成 |

### 4.3 异常处理
- **无法完成**：在任务卡对应位置写入问题描述（`tasks/{终端}/inbox/{任务卡编号}_blocked.md`），PM 看到后重新分配或调整
- **需要延期**：提前 24 小时通知 PM，写明原因和预计完成时间

## 5. 审核流程

### 5.1 审核标准
- 交付物是否完整（所有 checklist 项完成）
- 交付物是否正确（对照审核点逐项验证）
- 是否有新引入的问题（BUG、安全隐患）

### 5.2 审核结果
| 结果 | 操作 |
|------|------|
| ✅ 通过 | 写入 `tasks/archive/`，继续下一张任务卡 |
| 🔄 需修改 | 写明修改意见，发回 `inbox/` 重做 |
| ❌ 失败 | 写明失败原因，发回 `inbox/` 重新开始 |

### 5.3 审核原则
- PM 审核后才能归档，未审核不视为完成
- 审核结果 24 小时内给出（超时视为通过）

## 6. 反馈机制

### 6.1 正常反馈格式（写入 done/）
```markdown
# 完成反馈卡 DoneCard-{任务卡编号}
完成时间：YYYY-MM-DD HH:mm

## 任务编号
TC-XXX-{终端}

## 交付物确认
- [x] 交付物1：路径/说明
- [x] 交付物2：路径/说明

## 自检结果
[对照审核点自检，列出通过项]

## 遗留问题
[如有未完成项或风险，写明]

## 耗时
X 小时
```

### 6.2 异常反馈格式（写入 inbox/）
```markdown
# 异常反馈 BlockCard-{任务卡编号}
发现问题时间：YYYY-MM-DD HH:mm

## 任务编号
TC-XXX-{终端}

## 问题描述
[具体问题]

## 已尝试的解决方式
[列出]

## 需要 PM 支持的内容
[具体说明]
```

## 7. 归档规范

### 7.1 归档时机
- 审核通过后立即归档
- 每日 23:50 前完成当日任务卡归档

### 7.2 归档目录
```
tasks/{终端}/
├── inbox/          ← 待执行任务卡
├── done/           ← 已完成待审核
└── archive/        ← 已审核通过归档
    ├── 2026-05-XX/ ← 按日期归档
    │   ├── TC-001-xxx.md
    │   └── TC-002-xxx.md
    └── index.md    ← 归档索引（任务编号→文件路径）
```

### 7.3 归档索引格式
```markdown
# 归档索引 {YYYY-MM-DD}

| 任务编号 | 任务名称 | 完成时间 | 审核结果 | 审核时间 |
|----------|----------|----------|----------|----------|
| TC-001-niu | Day1 项目骨架 | 2026-05-06 14:00 | ✅通过 | 2026-05-06 15:30 |
```

## 8. PM 工作流程（日常）

### 8.1 每日节奏
| 时间 | 动作 |
|------|------|
| 随时 | 监听各端 inbox/done/，发现问题立即响应 |
| 每 10 分钟 | 检查 TASKS.md，查看任务进度 |
| 每日 23:50 | 各端提交当日进度，PM 汇总 |

### 8.2 PM 接收反馈后
1. 读取 `done/` 目录下的完成反馈卡
2. 对照审核点逐项验证
3. 写审核结果（通过/打回）
4. 归档或发回重做

### 8.3 PM 发出任务卡后
1. 在 TASKS.md 记录任务卡编号和预期完成时间
2. 等待各端反馈（inbox 或 done/）
3. 有异常立即响应，无异常等待

## 9. 质量门控

### 9.1 任务卡发出前（PM 自检）
- [ ] 任务卡格式完整（所有必填字段有值）
- [ ] 依赖关系正确（前序任务卡已完成才能开始）
- [ ] 交付物明确可验证（不是模糊描述）
- [ ] 与其他端关联已标注（接口依赖、WebSocket 等）

### 9.2 审核前（PM 自检）
- [ ] 对照审核点逐项检查
- [ ] 记录发现的问题
- [ ] 判断通过/打回/失败
