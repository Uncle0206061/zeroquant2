## 2. 项目基本信息
### 2.1 项目名称与版本
- 项目名称：ZeroQuant 2.0 A 股全自动量化交易系统
- 版本：MVP v2.2

### 2.2 项目背景与定位
- 前身：ZeroQuant 1.0 已归档
- 定位：面向 15 人小圈子的 B/S 架构 A 股全自动交易系统
- 模式：可视化策略编辑 → 数据筛选 → 多因子分析 → 模拟 / 实盘自动交易

### 2.3 核心价值
让有交易经验但无编程能力的用户，通过图形化界面完成量化策略编写与全自动交易。

### 2.4 运行环境与技术栈
| 层级 | 技术选型 |
|------|----------|
| 后端（妞妞） | Go 1.22+ / Gin（端口 8080） |
| 数据服务（数据官） | Python 3.10+ / FastAPI / akshare / Tushare Pro / Level-2（端口 8081） |
| 前端（大聪明） | Vue 3 + Vite + Element Plus（PC 端）+ Vant（移动端） |
| 存储 | PostgreSQL 15+、Redis |
| 部署 | Docker |

## 3. 技术规范

### 3.1 接口路由规范
所有 REST API 统一使用以下前缀，由三端 Day 1 约定确认：

| 前缀 | 说明 | 负责端 |
|------|------|--------|
| `/api/` | 业务接口（用户、策略、订单、持仓等） | 妞妞（Go） |
| `/data/` | 数据接口（行情、K 线、筛选结果等） | 数据官（Python） |

**响应格式规范（统一）：**
```json
{
  "code": 0,        // 0=成功，非0=失败
  "message": "success",     // 错误描述（失败时必填），成功时固定为 "success"
  "data": {}        // 业务数据（成功时有）
}
```
// ⚠️ ZERO.CC 修改：message 默认值从空改为 "success"，与技术规范 v2.2 统一

**错误码规范（5 类统一）：**
| 范围 | 说明 |
|------|------|
| 1xx | 系统级错误（DB/Redis/网络） |
| 2xx | 认证授权错误 |
| 3xx | 参数校验错误 |
| 4xx | 业务逻辑错误（余额不足、风控拦截等） |
| 5xx | 外部服务错误（数据源超时等） |

### 3.2 数据库规范
- PostgreSQL 数据库命名：小写 + 下划线（如 `zeroquant`）
- 三端共享同一个 PG 实例，不同 Schema 隔离
- **表前缀规范：**
  - `biz_` 前缀：业务数据表（用户、订单、持仓、策略），由 Go 后端管理
  - `data_` 前缀：数据存储表（行情、K 线、逐笔），由 Python 数据服务管理
- Redis Key 规范：
  - `biz:` 前缀：业务缓存（会话、持仓快照）
  - `data:` 前缀：数据缓存（实时行情、K 线）
- 所有主键使用 UUID（PostgreSQL 内置 uuid-ossp 扩展）

### 3.3 数据流路径（三类）

**① 实时行情数据流：**
```
外部数据源（akshare / Tushare Pro / Level-2）
  → Python 采集器（8081）
  → Redis（data: 缓存，TTL 5s）
  → WebSocket / HTTP
  → 前端实时展示
```

**② 策略执行数据流：**
```
用户前端编辑策略（表单 + 条件组合）
  → 提交至 Go 后端（/api/strategy/submit）
  → Go 策略引擎解析
  → 请求 Python 筛选服务（/data/filter）
  → Python 返回筛选结果
  → Go 执行撮合 / 订单逻辑
  → 结果写入 PG（biz_ 表）
  → WebSocket 推送前端
```

**③ 历史回测数据流：**
```
用户发起回测请求
  → Go 后端（/api/backtest）
  → Python 历史数据接口（/data/history）
  → Python 从 PG（data_ 表）取历史 K 线
  → 模拟撮合引擎执行回测
  → 返回收益 / 回撤 / 胜率等统计结果
```

### 3.5 Git 分支模型
```
main          ← 正式版本，只接受合并，禁止直接 push
dev           ← 开发主分支，每日合并 dev → dev
feat/{功能名}  ← 功能分支（各自开发用）
fix/{问题名}   ← 修复分支
doc/{文件名}   ← 文档分支（零号专用）
```

- **合并流程：** `feat/xxx` → `dev` → 审核 → `main`
- **提交频率：** 每人每日至少提交 1 次（23:50 前合并到 dev）
- **Commit 规范：** `[角色] 简要描述`，如 `[妞妞] 添加用户注册接口`

## 6. 整体开发阶段规划
### 6.1 阶段划分
1. **Phase 1**：MVP 核心闭环（模拟交易全流程可用）
2. **Phase 1.5**：实盘交易能力（券商对接、风控、安全机制）
3. **Phase 2**：策略增强（回测优化、策略对比、监控告警）
4. **Phase 3**：扩展能力（按需推进：多市场、高频特性）

### 6.2 各阶段任务与产出物
| 阶段 | 主要产出 |
|------|----------|
| Phase 1 | 三端骨架、数据服务、业务服务、前端页面、模拟交易闭环 |
| Phase 1.5 | 券商接口、实盘交易、风控、二次确认、紧急暂停 |
| Phase 2 | 回测增强、策略版本对比、操作建议、监控面板 |
| Phase 3 | 按需扩展 |

### 6.3 阶段依赖关系
- Phase 1.5 依赖 Phase 1 完成
- Phase 2 依赖 Phase 1.5 完成
- Phase 3 按需启动

## 8. 进度保障机制
### 8.1 每日进度同步
- 每日 **23:50 站会**（对齐代码提交时间），同步：进度、阻塞问题、风险

### 8.2 代码提交与合并规则
- 每日 23:50 前必须提交合并到 **dev** 分支
- 禁止直接 push 到 main 分支
- Commit 规范：`[角色] 简要描述`

### 8.3 联调与测试流程
后端自测 → 前后端联调 → 全链路验收 → 里程碑确认

### 8.4 变更管理流程
需求变更必须由 PM 统一确认，禁止私自调整范围与逻辑

|------|----------|------|

## 10. 质量与验收标准
### 10.1 功能验收标准
- 核心闭环 100% 可用
- 无阻塞 BUG、无数据错误
- 交互流程符合需求文档

### 10.2 性能验收标准
- 接口响应 95% ≤ 300ms
- WebSocket 推送延迟 ≤ 200ms
- 回测 1 年数据 ≤ 10 秒
- 数据误差 = 0%（双数据源校验）

### 10.3 安全与稳定性标准
- 敏感信息加密存储（AES）
- 实盘必须二次确认、风控自动拦截
- 异常自动降级、不崩溃、不丢数据

## 11. 风险识别与应对
### 11.1 技术风险
| 风险 | 应对 |
|------|------|
| Level-2 数据源不稳定 | 多源备份（akshare + Tushare Pro + Level-2），自动切换 |
| WebSocket 并发压力 | Redis 发布订阅、连接池管理 |
| 行情数据不准确 | 双数据源校验、日志可追溯 |

### 11.2 进度风险
| 风险 | 应对 |
|------|------|
| 联调阻塞 | 提前接口约定（Day 1），每日对齐 |
| 环境问题 | 统一版本、Docker 标准化 |
| M4 联调延期 | 加 2 天缓冲（05-12） |

### 11.3 业务风险
| 风险 | 应对 |
|------|------|
| 策略执行不准时 | 模拟盘充分测试 |
| 实盘风控失效 | 二次确认 + 一键暂停 + 自动冻结 |
